# Hermes Workspace VNC 镜像

> **仓库:** aicodewith-team/hermes-workspace-image (public)
> **产物:** Docker 镜像 `ghcr.io/aicodewith-team/hermes-workspace-vnc`
> **行数:** ~335 (Dockerfile 95 + entrypoint.sh 74 + start-vnc.sh 45 + nginx 配置 45 + CI 49 + README 111)
> **基础镜像:** `nousresearch/hermes-agent`（通过 SHA256 digest 固定，而非 tag）

## 功能简介

这是为需要 VNC 浏览器访问的 Hermes Agent 工作区准备的**预构建 Docker 镜像**。它在官方
`nousresearch/hermes-agent` 镜像的基础上扩展，将运行时所需的一切都内置进来——容器启动时无需执行
`apt-get`。控制平面在工作区 Nomad job 模板中通过 digest 引用此镜像；当工作区启动时，`entrypoint.sh`
是唯一的启动契约：生成 `config.yaml`、启动 `hermes-webui`、按需拉起 VNC 栈，然后以 PID 1 的身份
`exec` Hermes gateway。

**它不负责:** 运行时配置（环境变量来自控制平面的 Nomad 模板渲染）、持久化（挂载的卷）或网络路由
（由 Traefik 处理）。

## 技术栈

Docker（多阶段构建，SHA 固定基础镜像）· bash `entrypoint.sh`（set -euo pipefail）· Xvfb
（虚拟帧缓冲 :99）· x11vnc（构建时内置 Debian snapshot 库）· websockify（Python WebSocket→TCP
桥接，通过 `uv pip` 安装）· noVNC 1.5.0（静态 HTML5 客户端，无 `include/` 目录）· nginx（:80
前置入口，预配置）· hermes-webui（Gateway Bridge 模式，端口 :8787）· `/publish` StaticFiles 挂载
（构建时对 Hermes `web_server.py` 打补丁）· GitHub Actions CI（push 到 main 时构建并推送）

## 启动契约（`entrypoint.sh`）

`entrypoint.sh` 是**唯一的启动契约**——所有配置都来自环境变量，没有任何东西被硬编码进去：

1. **校验必需的环境变量** —— `HERMES_HOME`、`MODEL_ID`、`PROVIDER_NAME` 是必填的
   （通过 `:?` bash 参数展开）。`OPENAI_BASE_URL` 和 `API_SERVER_KEY` 有回退默认值。
2. **生成 `$HERMES_HOME/config.yaml`** —— 写入 `model.default`、`model.provider` 以及
   一个 `custom_providers` 列表项。`custom_providers` 下的格式是**列表**（而非字典），
   由一段 Python 代码校验，若不是列表则退出容器。
3. **启动 `hermes-webui`**（VNC 和非 VNC 模式均启动）—— 设置 `HERMES_WEBUI_HOST=0.0.0.0`
   和 `HERMES_WEBUI_GATEWAY_API_KEY=$API_SERVER_KEY`，在端口 8787 后台运行 `server.py`。
4. **条件式 VNC 栈** —— 若 `ENABLE_VNC=true`，调用 `start-vnc.sh` 并后台运行
   `hermes dashboard --port 9119`。
5. **以 PID 1 身份 `exec hermes gateway run --no-supervise`** —— 这个 `exec` 至关重要：
   没有它，bash 会吞掉 Nomad stop 发来的 `SIGTERM`，导致容器被 `SIGKILL`。

### 必需的环境变量

| 环境变量 | 必需 | 用途 |
|---|---|---|
| `HERMES_HOME` | ✅ | agent home 目录（`config.yaml` + `public/` 位于此处） |
| `MODEL_ID` | ✅ | 默认模型（例如 `claude-opus-4-7`） |
| `PROVIDER_NAME` | ✅ | 自定义 provider 名称（例如 `mergio-api`） |
| `OPENAI_BASE_URL` | — | LLM 端点的 provider `base_url` |
| `API_SERVER_KEY` | — | gateway API key（webui bridge 也会使用） |
| `ENABLE_VNC` | — | `true`/`false`，默认 `false` |

## VNC 栈（`start-vnc.sh`）

仅在 `ENABLE_VNC=true` 时调用。无 apt-get——一切均已在镜像中预安装：

1. **VNC 密码** —— 从 `/dev/urandom` 或 `$VNC_PASSWORD` 生成。写入
   `VNC_PASSWORD_FILE`（从 `$HERMES_HOME` 工作区根目录派生 → `vnc-password.txt`，
   与控制平面的读取路径一致）并 `chmod 600`。
2. **链接 noVNC** —— 将 `/opt/vnc/noVNC` 软链接到 `/opt/data/www/vnc`（nginx 以
   `/opt/data/www` 作为 web 根目录提供服务）。
3. **Xvfb** —— 在 `:99` 上运行虚拟帧缓冲，1280×720×24。
4. **x11vnc** —— `LD_LIBRARY_PATH` 指向内置的 Debian snapshot 库
   （`/opt/vnc/usr/lib/x86_64-linux-gnu` + `/opt/vnc/lib/x86_64-linux-gnu`），监听
   `:5900`，密码保护。
5. **websockify** —— WebSocket→TCP 桥接，`127.0.0.1:6080` → `127.0.0.1:5900`。
6. **nginx** —— 将 `user www-data` 改为 `user hermes`，把工作区配置软链接进
   `sites-enabled`，移除默认站点，在 `:80` 上启动 nginx。

## Nginx 路由（`nginx-workspace.conf`）

启用 VNC 时，nginx 监听 `:80` 并按如下路由：

| 路径 | 后端 | 协议说明 |
|---|---|---|
| `/websockify` | websockify `:6080` | WebSocket 升级，3600s 读超时 |
| `/webui/` | hermes-webui `:8787` | WebSocket 升级，3600s 读超时 |
| `/publish/` | Hermes dashboard `:9119` | 代理到 dashboard web server |
| `/` | 静态文件 → dashboard `:9119` | `try_files $uri $uri.html $uri/index.html @dashboard` 回退 |

当 `ENABLE_VNC=false` 时，没有 nginx——`hermes-webui` 和 `hermes gateway` 直接绑定到各自的
端口，由控制平面将它们映射到主机。

## `/publish` StaticFiles 挂载（构建时补丁）

上游 Hermes 的 `web_server.py` 没有 `/publish` StaticFiles 挂载。Dockerfile 在**构建时**
（而非容器启动时）为其打补丁，这样上游的结构性变更会让 CI 构建大声失败，而不是在每次启动时
静默地空操作。补丁插入如下内容：

```python
app.mount("/publish", StaticFiles(directory=get_hermes_home() / "public", html=True), name="publish")
```

插入位置在 `def mount_spa(application: FastAPI):` 之后。补丁通过 `grep` 检查验证；如果打补丁后
找不到该行，构建即失败。

## CI/CD（`build.yml`）

- **触发条件:** push 到 `main` 且修改了 `Dockerfile`、`entrypoint.sh`、`start-vnc.sh`、
  `nginx-workspace.conf` 或工作流本身。也支持手动 `workflow_dispatch`。
- **构建:** Docker Buildx，使用 GitHub Actions 缓存（gha 类型，mode=max）。
- **推送:** 到 `ghcr.io/aicodewith-team/hermes-workspace-vnc`，带三个 tag：
  `latest`、`webui` 和 `<git-sha>`（用于控制平面配置中的 digest 固定）。
- **权限:** `contents: read`，`packages: write`。

## 约定与陷阱

- **在 `Dockerfile` 中始终通过 SHA256 digest 固定 Hermes 基础镜像**，绝不要用 tag。
  Hermes 的 tag 是可变的，会导致构建不可复现。升级 Hermes 版本时更新 digest
  （见 README §Upgrading）。
- **`custom_providers` 必须是列表，而非字典。** `entrypoint.sh` 的 Python 校验器
  对此进行防护——字典曾在生产环境导致 "Unknown provider"。校验器若发现错误会以醒目的
  错误退出，使容器启动失败，而不是带着损坏的配置运行。
- **entrypoint.sh 中的 `exec` 至关重要。** 没有 `exec`，bash 会成为 PID 1 并吞掉
  `SIGTERM`。Nomad stop 随后会超时并发出 `SIGKILL`。始终把 gateway 作为最后一条命令 `exec`。
- **运行时不执行 `apt-get`。** 所有 VNC 依赖（x11vnc 库、nginx、Xvfb、x11vnc
  二进制、websockify、noVNC）都在构建时内置。运行时依赖放进 Dockerfile，
  而不是 Nomad 模板的启动脚本里。
- **x11vnc 库使用 Debian snapshot URL。** `libvncserver1` 和 `libvncclient1` 已从
  Trixie apt 中移除——Dockerfile 从带时间戳的 Debian snapshot 下载它们。升级到更新的
  Debian 基础镜像可能会破坏这一点；变更前请先测试。
- **noVNC 1.5.0 没有 `include/` 目录**（自 v1.4.x 起移除）。只复制 `app/`、`core/`、
  `vendor/` 和 `vnc.html`。升级 noVNC 版本可能改变目录结构——请核实复制步骤。
- **`/publish` 补丁在构建时验证。** 如果 `grep` 检查失败，构建会以错误退出。
  这是有意为之的——每次启动时静默空操作比一次响亮的构建失败更糟糕。
- **hermes-webui 在构建时 clone**，来源是 `github.com/nesquena/hermes-webui.git`
  （depth 1）。它使用 Gateway Bridge 模式——把浏览器聊天代理到 `127.0.0.1:8642` 上的
  Gateway API。由于它不直接 import `AIAgent`，所以没有 Agent 实例冲突。
- **`uv pip` 是包管理器**，不是 `pip`。Hermes venv 没有 pip 模块——
  使用 `/usr/local/bin/uv pip install --python /opt/hermes/.venv/bin/python3`。
- **nginx 以 `hermes` 用户运行**，而非 `www-data`。`start-vnc.sh` 在 nginx 启动时用
  `sed` 打补丁完成这一变更。
- **每次构建产生三个镜像 tag:** `latest`（浮动）、`webui`（用于 webui 专用引用）、
  `<git-sha>`（控制平面配置的稳定固定目标）。生产环境中始终引用 SHA tag。
