# Hermes Workspace VNC Image

> **Repo:** aicodewith-team/hermes-workspace-image (public)
> **Artifact:** Docker image `ghcr.io/aicodewith-team/hermes-workspace-vnc`
> **Lines:** ~335 (Dockerfile 95 + entrypoint.sh 74 + start-vnc.sh 45 + nginx config 45 + CI 49 + README 111)
> **Base:** `nousresearch/hermes-agent` (pinned by SHA256 digest, not tag)

## What It Does

This is the **pre-baked Docker image** for Hermes Agent workspaces that require VNC browser
access. It extends the official `nousresearch/hermes-agent` image with everything needed at
runtime — no `apt-get` at container boot. The control plane references this image (by digest)
in workspace Nomad job templates; when a workspace launches, `entrypoint.sh` is the single
startup contract: generate `config.yaml`, start `hermes-webui`, optionally fire up the VNC
stack, then `exec` the Hermes gateway as PID 1.

**It does NOT own:** runtime configuration (env vars come from the control plane's Nomad
template rendering), persistence (mounted volumes), or network routing (Traefik handles that).

## Tech Stack

Docker (multi-stage, SHA-pinned base) · bash `entrypoint.sh` (set -euo pipefail) · Xvfb
(virtual framebuffer :99) · x11vnc (Debian snapshot libraries baked at build time) ·
websockify (Python WebSocket→TCP bridge, installed via `uv pip`) · noVNC 1.5.0 (static
HTML5 client, no `include/` directory) · nginx (:80 front door, pre-configured config) ·
hermes-webui (Gateway Bridge mode, port :8787) · `/publish` StaticFiles mount (build-time
patch to Hermes `web_server.py`) · GitHub Actions CI (build + push on push to main)

## Startup Contract (`entrypoint.sh`)

`entrypoint.sh` is the **single startup contract** — all configuration comes from environment
variables, nothing is baked in:

1. **Validate required env vars** — `HERMES_HOME`, `MODEL_ID`, `PROVIDER_NAME` are mandatory
   (`:?` bash parameter expansion). `OPENAI_BASE_URL` and `API_SERVER_KEY` have fallback defaults.
2. **Generate `$HERMES_HOME/config.yaml`** — writes `model.default`, `model.provider`, and
   a `custom_providers` list entry. The format is a **list** (not dict) under `custom_providers`,
   validated by a Python snippet that exits the container if it's not a list.
3. **Start `hermes-webui`** (both VNC and non-VNC modes) — sets `HERMES_WEBUI_HOST=0.0.0.0`
   and `HERMES_WEBUI_GATEWAY_API_KEY=$API_SERVER_KEY`, backgrounds `server.py` on port 8787.
4. **Conditional VNC stack** — if `ENABLE_VNC=true`, calls `start-vnc.sh` and backgrounds
   `hermes dashboard --port 9119`.
5. **`exec hermes gateway run --no-supervise`** as PID 1 — the `exec` is load-bearing:
   without it, bash swallows `SIGTERM` from Nomad stop and the container gets `SIGKILL`'d.

### Required Environment Variables

| Env | Required | Purpose |
|---|---|---|
| `HERMES_HOME` | ✅ | agent home directory (`config.yaml` + `public/` live here) |
| `MODEL_ID` | ✅ | default model (e.g., `claude-opus-4-7`) |
| `PROVIDER_NAME` | ✅ | custom provider name (e.g., `mergio-api`) |
| `OPENAI_BASE_URL` | — | provider `base_url` for LLM endpoint |
| `API_SERVER_KEY` | — | gateway API key (also used by webui bridge) |
| `ENABLE_VNC` | — | `true`/`false`, defaults `false` |

## VNC Stack (`start-vnc.sh`)

Called only when `ENABLE_VNC=true`. No apt-get — everything is pre-installed in the image:

1. **VNC password** — generated from `/dev/urandom` or `$VNC_PASSWORD`. Written to
   `VNC_PASSWORD_FILE` (derived from `$HERMES_HOME` workspace root → `vnc-password.txt`,
   matching the control plane's read path) and `chmod 600`.
2. **Link noVNC** — symlinks `/opt/vnc/noVNC` → `/opt/data/www/vnc` (nginx serves from
   `/opt/data/www` as web root).
3. **Xvfb** — virtual framebuffer on `:99`, 1280×720×24.
4. **x11vnc** — with `LD_LIBRARY_PATH` pointing to the baked Debian snapshot libraries
   (`/opt/vnc/usr/lib/x86_64-linux-gnu` + `/opt/vnc/lib/x86_64-linux-gnu`), listening
   on `:5900`, password-protected.
5. **websockify** — WebSocket→TCP bridge on `127.0.0.1:6080` → `127.0.0.1:5900`.
6. **nginx** — patches `user www-data` → `user hermes`, symlinks the workspace config
   into `sites-enabled`, removes the default site, starts nginx on `:80`.

## Nginx Routing (`nginx-workspace.conf`)

When VNC is enabled, nginx listens on `:80` and routes:

| Path | Backend | Protocol Notes |
|---|---|---|
| `/websockify` | websockify `:6080` | WebSocket upgrade, 3600s read timeout |
| `/webui/` | hermes-webui `:8787` | WebSocket upgrade, 3600s read timeout |
| `/publish/` | Hermes dashboard `:9119` | proxied to dashboard web server |
| `/` | static files → dashboard `:9119` | `try_files $uri $uri.html $uri/index.html @dashboard` fallback |

When `ENABLE_VNC=false`, there is no nginx — `hermes-webui` and `hermes gateway` bind
directly to their ports and the control plane maps them to the host.

## `/publish` StaticFiles Mount (Build-Time Patch)

The upstream Hermes `web_server.py` has no `/publish` StaticFiles mount. The Dockerfile patches
this **at build time** (not container startup) so a structural change upstream fails the CI build
loudly instead of silently no-opping on every boot. The patch inserts:

```python
app.mount("/publish", StaticFiles(directory=get_hermes_home() / "public", html=True), name="publish")
```

after `def mount_spa(application: FastAPI):`. The patch is verified with a `grep` check; if the
line isn't found post-patch, the build fails.

## CI/CD (`build.yml`)

- **Trigger:** push to `main` that modifies `Dockerfile`, `entrypoint.sh`, `start-vnc.sh`,
  `nginx-workspace.conf`, or the workflow itself. Also manual `workflow_dispatch`.
- **Build:** Docker Buildx with GitHub Actions cache (gha type, mode=max).
- **Push:** to `ghcr.io/aicodewith-team/hermes-workspace-vnc` with three tags:
  `latest`, `webui`, and `<git-sha>` (for digest pinning in the control plane config).
- **Permissions:** `contents: read`, `packages: write`.

## Conventions & Pitfalls

- **Always pin the Hermes base image by SHA256 digest in `Dockerfile`**, never by tag.
  Hermes tags are mutable and cause non-reproducible builds. Update the digest when
  upgrading Hermes versions (see README §Upgrading).
- **`custom_providers` must be a list, not a dict.** The `entrypoint.sh` Python validator
  guards against this — a dict caused "Unknown provider" in production. The validator
  exits with a loud error if it's wrong, so the container fails to start rather than
  running with a broken config.
- **`exec` in entrypoint.sh is load-bearing.** Without `exec`, bash becomes PID 1 and
  swallows `SIGTERM`. Nomad stop then times out and issues `SIGKILL`. Always `exec` the
  gateway as the final command.
- **No `apt-get` at runtime.** All VNC dependencies (x11vnc libs, nginx, Xvfb, x11vnc
  binary, websockify, noVNC) are baked at build time. Runtime deps go in the Dockerfile,
  not in Nomad template startup scripts.
- **x11vnc libraries use Debian snapshot URLs.** `libvncserver1` and `libvncclient1` were
  removed from Trixie apt — the Dockerfile downloads them from a timestamped Debian
  snapshot. Upgrading to a newer Debian base may break this; test before changing.
- **noVNC 1.5.0 has no `include/` directory** (removed since v1.4.x). Only `app/`, `core/`,
  `vendor/`, and `vnc.html` are copied. Upgrading noVNC versions may change the directory
  structure — verify the copy step.
- **`/publish` patch is verified at build time.** If the `grep` check fails, the build
  exits with error. This is intentional — silent no-ops on every boot are worse than a
  loud build failure.
- **hermes-webui is cloned at build time** from `github.com/nesquena/hermes-webui.git`
  (depth 1). It uses Gateway Bridge mode — proxies browser chat to the Gateway API on
  `127.0.0.1:8642`. No Agent instance conflict because it doesn't import `AIAgent` directly.
- **`uv pip` is the package manager**, not `pip`. The Hermes venv has no pip module —
  use `/usr/local/bin/uv pip install --python /opt/hermes/.venv/bin/python3`.
- **nginx runs as `hermes` user**, not `www-data`. `start-vnc.sh` patches this at nginx
  startup with `sed`.
- **Three image tags on every build:** `latest` (floating), `webui` (for webui-specific
  references), `<git-sha>` (the stable pin target for control plane config). Always
  reference the SHA tag in production.
